### Potsdam 数据集替换决策

本轮重构将默认训练数据从 DOTA 切换为 Potsdam。原因是 Potsdam 正射影像分辨率更高，适合用“原图截图/裁剪”来模拟无人机悬停拍摄视角；DOTA 旧流程仍保留为兼容选项。

实现策略：
- 数据入口使用 `--dataset potsdam`，扫描 `train/images + train/labels` 或 ISPRS 原始 `2_Ortho_* / 3_Ortho_* / 5_Labels_*` 目录。
- Potsdam 语义 mask 经连通域分析转为伪实例，实例结构与原 DOTA OBB 解析结果保持一致：`class/cx/cy/poly/difficult`。因此后续 `sample_episode`、`generate_trajectory`、截图裁剪、ShareGPT JSONL 输出基本复用。
- 类别映射改为统一 `DATASET_CATEGORY_MAP`，新增 `car/building/tree/low-vegetation/impervious-surface`，其中 `car` 作为目标类，`building` 可同时作目标和地标，植被/硬化地面主要作地标。
- Potsdam 使用 `POTSDAM_GSD=0.05m/px`，不走 DOTA 的 OBB 物理先验反推 GSD。评估端若传入 raster label，也使用该 GSD 生成截图视野。
- 运动学、STOP 阈值、历史动作、累计位移、Turn-then-Move 模型保持不变。这部分本质是“截图模拟无人机飞行”的核心，不依赖 DOTA。

### 1. 完整架构定义
#### 1.1 坐标与方向严格定义
为了防止模型混淆，必须在System Prompt中严格定义坐标系：
*   **Yaw（方向）定义**：正北为 $0^\circ$，顺时针为正（东=$90^\circ$，南=$180^\circ$，西=$270^\circ$）。
*   **网格坐标系定义**：将无人机当前视野划分为 $11 \times 11$ 网格。
    *   **原点 $(0,0)$**：为无人机正下方（视野中心）。
    *   **X轴（左右）**：无人机面朝正前方时，右侧为X正方向，左侧为X负方向。范围 $[-5, 5]$。
    *   **Y轴（前后）**：正前方为Y正方向，正后方为Y负方向。范围 $[-5, 5]$。
#### 1.2 输入定义
*   **Task (任务)**：自然语言指令，如“去教学楼旁边的红色汽车处”。
*   **Current Image (当前图像)**：无人机当前视角的图像。
*   **History Images (历史图像)**：考虑到大模型（如Qwen2.5-VL-7B）处理多张高分辨率图像极易导致显存溢出（OOM），建议**只保留1张当前图像和1张上一帧（T-1）图像**。
*   **History Actions (历史动作)**：模型在过去 5 步输出的局部网格坐标序列（如 `[(0, 5), (3, 4), (-1, 4), (2, 3), (0, 2)]`），让模型感知近期运动节奏。保留 5 步是因为大尺度导航需要识别"已经走了多远 / 是否在绕路"等时序模式，2 步过短。
*   **Cumulative State (累计状态向量)**：从起点开始的累计**全局位移**（地理北/东向），**单位为"格"（无量纲），不是米**。例如：`累计位移: 向北12.4格, 向东5.1格, 已飞5步`。
    *   **为什么用"格"而不是"米"**：本地 DOTA 数据 99% 没有真实 gsd（见 §4.1 实测），任何用"米"表达的累计位移都依赖虚构的 `grid_size_m`，数字本身是假的。改用"格"后单位与 `Action: (x, y)` 完全一致，**消除所有虚构物理量**。
    *   **为什么是"全局视角"而非"无人机视角"**：北/东两个分量始终指**地理北、地理东**（DOTA 图像 -y / +x 方向），与无人机当前朝向无关。即使无人机一直转弯，"向北 12 格"始终指真实地理北方。公式中的 $\sin(\text{yaw})/\cos(\text{yaw})$ 就是把每一步的局部步长投影到全局北/东。
    *   **数据来源**：脚本同步维护 yaw 序列和 action 序列，按以下公式积分得到全局网格位移（**无 grid_size 乘子**）：
        - $\Delta E_{global} = \sum_t |\text{action}_t| \cdot \sin(\text{yaw}_t)$ （东向，Yaw=90° 时为正）
        - $\Delta N_{global} = \sum_t |\text{action}_t| \cdot \cos(\text{yaw}_t)$ （北向，Yaw=0° 时为正）
        - 其中 $|\text{action}_t| = \sqrt{\text{gx}_t^2 + \text{gy}_t^2}$ 是该步沿（更新后）朝向直行的网格数。
        - **yaw 时序约定**：积分时用的 $\text{yaw}_t$ 必须是**该步物理更新后的 yaw**（即 $\text{yaw}_t = \text{yaw}_{t-1} + \arctan2(\text{gx}, \text{gy}) + \text{noise}$ 之后的值），与 Turn-then-Move 真实位移方向一致。
    *   **实机部署对齐**：飞控 IMU 给出真实米数 → 除以机载实时算出的 `grid_size_m`（由飞行高度 + 相机 FOV 决定）→ 得到全局网格数喂模型。训练-评估-实机三方单位严格统一。
    *   **关键约束**：yaw 是飞控自带的相对量（任何无人机都有磁罗盘），不属于 GPS "特权信息"；不使用 DOTA 像素坐标或 GPS 经纬度，保证 sim-to-real 严格一致。
    *   **作用**：替代视觉里程计 (VO)，让模型在悬停拍摄过程中拥有真正的全局位置感，避免无限绕路。
    *   **避坑提示**：早期方案曾用"前进/右移代数和"，在 yaw 变化时会累出错误数字（先北飞 5 格再东飞 5 格会累出"前进 10 格, 右移 0 格"），实际位置却是东北方向 (北 5 格, 东 5 格)。务必用 yaw 积分而非局部动作量求和。

#### 1.3 输出定义
**强烈建议输出结构化文本，不要直接输出纯数字，也不要用1-121的1D索引。**
格式设计如下：
```text
Thought: 观察当前图像，正前方有一栋教学楼，右侧有一辆红色汽车，符合目标描述。当前距离地标较近，红色汽车在视野的右前方。
Action: (3, 4)  # X=3(右侧), Y=4(前方)
```

**STOP 决策（训练-评估对齐的统一阈值）**：
- 当无人机距离最终目标 $\leq \max(5\text{m},\, 0.5 \times \text{grid\_size})$ 时，必须输出 STOP（与 §3.2 SR 评估阈值完全相同）。
- **`Action: STOP` 与 `Action: (0, 0)` 在解析层完全等价**，飞控/评估脚本一并处理。模型可输出任意一种格式，下游脚本统一映射为停止。
- 训练数据中两种表达按 **80% (STOP) : 20% (0,0)** 比例混合采样标签，保证模型对两种格式都能稳定输出；评估推理时若正则解析失败也兜底为 STOP，避免无限循环。
- 这套阈值同时是训练标签的 STOP 边界和评估的成功边界，**彻底消除了"训练学到要更近才停 / 评估算更远也成功"的分布偏移**。
#### 1.4 完整Prompt模板
*注：这里输入格式使用的是结构化纯文本（如 `任务: xxx\n当前状态: xxx`）而不是 JSON 格式（如 `{"task": "xxx", "state": {"yaw": 45}}`）。这是因为大语言模型（如Qwen）在预训练和指令微调阶段接触的绝大多数多轮对话都是人类自然语言格式。结构化纯文本不仅占用 Token 更少，而且更符合 LLM 的自然语义理解习惯，表现通常优于强行包裹成 JSON 的输入格式。*

```text
[System]
你是一个无人机视觉语言导航助手。你的任务是根据指令和视野，输出下一步的移动网格坐标。
基本设定：正交俯视视角，每张图像已经按当前朝向旋转——画面正上方 = 你的正前方。
坐标系定义：视野为 11x11 网格，中心为(0,0)。X轴：右正左负(-5到5)；Y轴：前正后负(-5到5)。Yaw 角：正北 0 度，顺时针为正。
累计位移坐标系：北/东为地理方向（DOTA 图像 -y / +x 方向），与你当前朝向无关；单位是"格"（与 Action 单位一致）。
输出格式：
Thought: 简短分析当前观测与目标的关系
Action: (x, y) 或 STOP
停止规则：当目标进入视野中心半个网格内时，输出 Action: STOP。Action: STOP 与 Action: (0, 0) 在系统解析中完全等价，二选一即可。
[User]
<image>
<image>
任务: 去{地标}旁边的{目标}。
历史动作: [(0, 5), (3, 4), (-1, 4), (2, 3), (0, 2)]
累计位移: 向北12.4格, 向东5.1格, 已飞5步
请输出下一步动作。
```

### 2. 训练数据格式与框架
#### 2.1 数据格式 (JSONL)
采用 LLaMA-Factory 标准 ShareGPT 多模态格式。**注意字段名**：经典 ShareGPT 使用 `"from"` / `"value"` 字段（不是 OpenAI 的 `role` / `content`），且 `<image>` 占位符数量必须严格等于 `images` 数组长度，否则 LLaMA-Factory 加载时会报错。

```json
{
  "conversations": [
    {
      "from": "human",
      "value": "<image>\n<image>\n任务: 去主楼旁边的雷达天线。\n历史动作: [(1, 5), (0, 4), (2, 5), (1, 3), (0, 2)]\n累计位移: 向北14.0格, 向东4.0格, 已飞5步\n请输出下一步动作。"
    },
    {
      "from": "gpt",
      "value": "Thought: 当前视野正前方是主楼，雷达天线在主楼右侧，即我的右前方方向。\nAction: (3, 4)"
    }
  ],
  "images": [
    "/path/to/step1_img_hist.jpg",
    "/path/to/step2_img_current.jpg"
  ]
}
```

并在 `data/dataset_info.json` 中注册

**单阶段全量训练原则（重要）**：
本方案采用**单阶段 SFT**，所有字段（2 张图 + 历史动作 + 累计位移 + CoT + Action）一次性全量训练。**不再做"Stage 1 mask 字段 + Stage 2 放开"的两阶段课程**——经评估，该设计在第 N 步图像上灌入零历史/零累计位移会教会模型"忽略这两个字段"，与 Stage 2 目标直接冲突（反向 curriculum）。

理由：
1. Qwen2.5-VL-7B 是已 instruction-tuned 的 VLM，对 `Thought: ... Action: (x,y)` 这种结构化输出在 200-500 步即可收敛，无需专门一阶段对齐格式。
2. 50k-100k 样本 + LoRA 是标准 SFT 规模，一阶段足够；省掉额外 1× 训练时间和 LoRA continual learning 的灾难性遗忘风险。
3. 若仍想要 Curriculum，应通过**数据采样顺序**（先短距/STOP，后远距探索）在单阶段内实现，而不是按字段 mask。
4. `<image>` 占位符数量与 images 数组长度始终对齐（2:2），避免 LLaMA-Factory 加载报错。
#### 2.2 训练框架与方式
*   **框架**：**LLaMA-Factory**（目前对Qwen-VL适配最好、最易用的开源微调框架，极力推荐）。
*   **训练方式**：**LoRA / QLoRA微调**。用LoRA在两张4090上可以轻松完成，单张4090也能跑（需适当降低分辨率或Batch Size）。
*   **关键参数**：
    *   `lora_rank`: 64 或 128
    *   `lora_target`: 除Vision Encoder外的所有Linear层（如 `q_proj,k_proj,v_proj,o_proj`）
    *   `resolution`: 448x448 或 896x896（视野清晰度很重要，显存够尽量开大）

### 3. 基准与指标
#### 3.1 基准定义
你需要对比你的微调模型与以下基线：
1.  **Random**：在11x11网格中随机输出合法坐标。
2.  **Zero-shot Qwen2.5-VL-7B**：不微调，直接用Prompt问原模型要坐标。
3.  **Rule-based (Greedy)**：传统贪心算法，每次计算目标在视野中的方位，直接输出该方位的最大步长坐标。
    *注：在DOTA这种高空俯视场景下，基本假设为“无障碍空间”，因此弱化甚至不考虑物理避障，重点评估模型的“空间方位推理”和“指令理解”能力。*

#### 3.2 评估指标
1.  **成功率 (SR)**：在最大步数（如20步）内，无人机中心进入目标周围 $\epsilon$ 米半径内即视为成功。
    *   **多高度自适应阈值**：固定 2m 在 100m 高度根本到不了；固定网格数掩盖了高高度的物理精度劣势。采用如下公式：
        $$\epsilon = \max(\epsilon_{min}, \, \alpha \times \text{grid\_size})$$
        其中 $\epsilon_{min}=5m$（实机 GPS / RTK 精度下限），$\alpha=0.5$（半个网格内算成功）。
    *   **训练-评估对齐**：训练数据生成时的 STOP 阈值必须使用**完全相同**的 $\epsilon$ 公式（见 §1.3），杜绝"训练学到要更近才停 / 评估算更远也成功"的分布偏移。
    *   **示例**：
        | 飞行高度 | grid_size | $\epsilon$ |
        | --- | --- | --- |
        | 50m | 10m | 5m |
        | 100m | 20m | 10m |
        | 150m | 30m | 15m |
    *   **报告时按高度档位分别给 SR**（SR@50m / SR@100m / SR@150m），避免用单一数字掩盖高度差异；也可以再单独报一个 "归一化 SR" = 是否进入 $\alpha$ 网格内，便于跨高度比较。
2.  **SPL (Success weighted by Path Length)**：VLN核心指标。成功率和路径效率的加权，惩罚绕路的成功轨迹。
    $$SPL = \frac{1}{N} \sum_{i=1}^N S_i \frac{l_i}{\max(p_i, l_i)}$$
    （$S_i$为是否成功，$l_i$为最短路径，$p_i$为实际路径）
3.  **平均步数**：成功轨迹中达到目标所需的平均步数。

### 4. 数据集生成与仿真脚本设计（核心工程）
#### 4.1 多高度仿真与视场映射 (FOV Mapping)

> **【单位本质澄清】**
> 本项目所有"米"是基于 `FIXED_GSD=0.5 m/px` 的**评估端基准假设值**,DOTA 卫星图本身无可靠 GSD。
> 真正驱动模型学习的物理量是**像素**:
>   - `crop_size` (px) 决定视野大小
>   - `grid_pixel` (px) 决定 STOP 半径
>   - 起点像素距离 决定 episode 难度
> "米"仅作为内部 unit consistency 用,prompt 中不暴露(高度/grid_size/距离)。
> 模型输出无量纲 grid (x,y);实机部署时飞控用**真实 GSD**(由相机内参+飞行高度算出)换算成米——训练假设的 GSD 与实机真实 GSD 不需要严格对应。
> **训练端额外做 GSD 抖动数据增强**(每条轨迹 `GSD ∈ [0.35, 0.7]`),让模型对绝对像素尺度更鲁棒;**评估端固定 GSD=0.5** 保证 SR/SPL 复现性。详见 §4.1 "训练 GSD 抖动 + 评估固定 GSD" 小节。

为了支持不同高度的导航，需要建立"飞行高度 -> 视场大小 -> 裁剪像素尺寸"的映射关系。
*   **物理换算**：假设无人机相机视场角(FOV)为 $90^\circ$，则地面视野边长 $L = 2 \times H \times \tan(45^\circ) = 2H$。
    *   高度 $H=50m$ 时，视野 $L=100m$。网格体系（$11 \times 11$）中每个网格对应 $\approx 10m$。
    *   高度 $H=100m$ 时，视野 $L=200m$。网格体系（$11 \times 11$）中每个网格对应 $\approx 20m$。
*   **图像裁剪换算（训练 GSD 抖动 + 评估固定 GSD）**：
    *   **本地数据事实（实测）**：DOTA v2.0 本地 419 张 train 图中，**98.8% 的 meta 文件标注 `gsd:None`**，仅 5 张数值（全为 1.0）；val 135 张中 94.8% 为 None，7 张数值范围 0.1–0.81。这意味着**早期"按图读 gsd 动态裁剪"的设计在本地数据上完全失效**（99% 图都走 default 兜底，相当于一刀切）。
    *   **修订策略：训练端 GSD 抖动 + 评估端固定 GSD**，删除"过滤 GSD ∉ [0.1, 1.5]"和"按图动态裁剪"逻辑。
        - **训练数据生成端**：每条轨迹独立抽样 `GSD ∈ [0.35, 0.7]`（`ppm ∈ [1.43, 2.86]`）。`crop_size = fov_m × ppm` 自动随之变化（同一 `fov_m=100` 在 GSD=0.35 下是 285 px 视野、在 GSD=0.7 下是 143 px 视野），让模型见过不同绝对像素尺度下的同一语义任务，缓解 sim-to-real 时实机 GSD 偏离训练假设带来的性能塌方。
        - **评估端 / 默认**：固定 `GSD = 0.5 m/px`（`ppm = FIXED_PPM = 2.0`），保证 SR/SPL 复现性。评估端**绝不抖动 GSD**——否则同一 episode 跑两次得到不同 SR，数字无意义。
        - **关键调用契约（违反会导致训练数据彻底污染）**：`pixel_per_meter(gsd_jitter=...)` 必须由顶层入口（`batch_generate` / `run_evaluation_episode`）在轨迹入口调用**一次**，把返回值显式传给所有下游函数（`sample_episode` / `compute_instruction_context_at_start` / `generate_trajectory` / `crop_uav_view`）。严禁下游函数内部再次调用 — 会让起点像素位置 / D 场景视野判定 / 实际裁剪三者使用不同的视野尺度，训练数据语义错位。
    *   **理由（三联强化）**：(a) 99% 的 meta 没有可信 gsd 字段，启发式/视觉反推都不可靠；(b) Qwen2.5-VL 是语义导航模型，对物理尺度的微小差异有相当的鲁棒性，模型主要靠"目标 vs 视野"的语义关系做决策；(c) 仿真训练 → 实机部署时实机 gsd 由飞行高度 + 相机 FOV 决定，本就不依赖训练阶段的 gsd 数值——GSD 抖动反而让模型对实机真实 GSD 偏离 0.5 这一基准更鲁棒。
    *   **保留 meta 读取作为元信息**：仍读取 `meta/{P-id}.txt`，但只把数值 gsd 当作 trajectory 元信息记录，不用于动态裁剪；trajectory 字段 `gsd_used` 记录该轨迹**实际抽样后**的 `1/ppm`（训练端为抖动值、评估端固定 0.5），便于事后数据质量分析。
    *   **裁剪换算（评估端基准公式）**：`ppm = 1 / 0.5 = 2 px/m`；高度 50m → 视野 100m → crop_size = 100 × 2 = 200 px；高度 100m → 视野 200m → crop_size = 400 px。裁剪后统一 Resize 到 `IMG_SIZE`（336 或 448）。**训练端因 GSD 抖动，crop_size 会按 ppm ∈ [1.43, 2.86] 等比例缩放（约 ±40%），但 Resize 到 IMG_SIZE 后送入模型的张量尺寸完全一致。**
    *   **后续路径**：若后期获得真实实机数据（PLAN §7 Stretch Goal），其 gsd 由实机相机参数算出，那一步再做细化。
    *   **裁剪示例（评估端基准 gsd=0.5, ppm=2）**：
        | 飞行高度 | 视野边长 | crop_size (评估) | crop_size 训练抖动范围 | grid_size |
        | --- | --- | --- | --- | --- |
        | 50m  | 100m | 200 px | 143 - 285 px | 10m/网格 |
        | 100m | 200m | 400 px | 286 - 571 px | 20m/网格 |
        | 150m | 300m | 600 px | 429 - 857 px | 30m/网格 |
        裁剪后统一 Resize 到 `IMG_SIZE` 送入模型（见 §5.5 的 GPU 配置推荐）。

##### 4.1.1 多高度训练数据采样策略（关键，对齐 §3.2 评估要求）

§3.2 强制要求按高度档位分别报告 `SR@50m / SR@100m / SR@150m`，故训练数据必须**覆盖三个高度档位**，否则评估期会出现严重的分布外（OOD）问题（训练只见过 50m、评估却要测 100/150m → SR@100m/150m 数字无意义）。

**采样比例**（每条轨迹生成前先采样高度档位，再带入对应 `grid_size_m` / `fov_m`）：

| 高度档位 | 占比 | grid_size_m | fov_m | crop_size (ppm=2) | 起点距离档位 |
| --- | --- | --- | --- | --- | --- |
| 50m  | **50%** | 10 | 100 | 200 px | A:[5,15m] / B:[30,60m] / C:[70,120m] / E:≤10m |
| 100m | **30%** | 20 | 200 | 400 px | A:[10,30m] / B:[60,120m] / C:[140,240m] / E:≤20m |
| 150m | **20%** | 30 | 300 | 600 px | A:[15,45m] / B:[90,180m] / C:[210,360m] / E:≤30m |

**关键设计要点**：

1. **比例倾斜的理由**：低空 50m 视野适中、grid 数足、最容易稳定收敛，且实机部署最常用，故占主导（50%）。100m / 150m 各保留一定比例以验证多尺度泛化能力。
2. **起点距离按高度等比例放大**：50m 场景的 `[5, 120m]` 起点分布在 100m 场景必须缩放到 `[10, 240m]`，否则低高度的"近距样本"在 100m 视野下会变成"零距样本"（目标距离远小于 grid_size），STOP 决策失效。
3. **高度信息不进 prompt（关键修订）**：DOTA 卫星图本身没有"飞行高度"概念，把虚构的"50m / 100m / 150m"标签喂给模型，等于教模型相信一个**无视觉验证手段**的伪标签。多高度信号**隐式存在于像素中**——同一架飞机在 50m 视野下占 ~30 像素，在 100m 视野下经过 resize 仅占 ~15 像素，模型可从目标的相对像素大小自行推断尺度。
4. **累计位移单位"格"（无量纲）天然支持多高度**（详见 §1.2）：50m 场景每步最多走 5 格、100m 场景每步也最多走 5 格——单位是相对网格而非绝对米，多高度数据在 prompt 数字范围上完全一致，避免模型困惑。
5. **STOP 阈值的多高度自适应**：训练标签 STOP 半径 `max(5m, 0.5 × grid_size_m)` 在数据生成端（脚本侧）按对应高度算，但 prompt 不暴露 grid_size——模型只需要学到"目标在视野中心半格内 → STOP"这一**纯视觉判定**（见 §1.3 / §3.2）。

#### 4.2 脚本逻辑：DOTA 类别 → 自然语言指令映射

DOTA v1.5 / v2.0 共 18 个类别（英文标签如 `small-vehicle`），无法直接出现在自然语言指令中。需要建立"类别角色 + 中文短语"映射表，脚本据此自动构造形如 `"去 {landmark_zh} 旁边的 {target_zh}"` 的指令。

> **L** = Landmark（地标，大型显著、单/少实例）；**T** = Target（目标，小型密集多实例）。

##### 4.2.2 指令构造算法

1. **选目标**：从图像内所有标注中，按"角色=T 或 L/T"的池子随机选 1 个作为最终目标 `target`。
2. **选地标**：在 target 周围 80m（≈ 8 网格）半径内，按"角色=L 或 L/T"的池子选**距离最近的** 1 个作为 `landmark`（如果没有，则放宽到 200m）。
3. **构造短语（按 target 角色 / landmark 存在性分支）**：
   - **有 landmark**：`"去{landmark_zh}旁边的{target_zh}。"`（标准形）
   - **无 landmark 且 target 自身可作地标（L/B 类）**：`"去{target_zh}。"`（简化形）
   - **无 landmark 且 target 是纯 T 类（长尾兜底，关键）**：必须用 `"去画面{spatial_word}的{target_zh}。"`（消歧形）。原因：本地 DOTA 数据 `small-vehicle` 占 63%，多数城市俯视图根本没有 L 类地标，若仍用 `"去{target_zh}"` 兜底会让指令彻底丢失"参照系"语义，模型在多目标场景下无法泛化。spatial_word 由调用方根据起点视野下 target 相对当前视野中心的方位（左上 / 右下 / 正前等）预先计算。
   - **触发显式消歧**：若同类物体数 > 1，无论 landmark 是否存在都强制走消歧形。
4. **同类计数 → 触发消歧**：如果当前视野内 target 类别的物体数 > 1，必须用"带消歧形"，否则模型无法区分。

##### 4.2.3 空间词消歧（沿用原方案）

当视野中存在多个相同类别的目标时，脚本需自动生成"空间限定词"进行消歧。
1.  **全局搜索**：读取 DOTA 标注，找出所有与目标类别相同的物体集合 $S$。
2.  **聚类与判断**：如果以当前视野为中心，一定半径内 $S$ 中的物体数量 $>1$，则触发空间词生成。
3.  **计算相对方位**：将目标位置 $(X_{target}, Y_{target})$ 转换到以当前视野范围内的局部坐标系。
4.  **分配形容词**：
    *   依据象限分配："画面左上方"、"画面右下方"的 {类别}。
    *   依据距离分配：计算无人机距离 $S$ 中各物体的距离，选取"距离最近的"或"距离最远的"。
    *   *备用方案（视觉标记）*：若纯文本难以消歧，脚本利用 OpenCV 在目标物体的中心坐标画一个红圈，并将指令改为"飞向红圈标注的 {类别}"。

#### 4.3 脚本逻辑：轨迹生成与物理噪声注入 (Turn-then-Move)
为了贴近真实无人机的运动学约束，采用**Turn-then-Move（先旋转对准，再直线飞行）**的贪心寻路逻辑，并**引入运动噪声（数据增强）**以提升模型对真实环境中风偏、惯性和控制误差的鲁棒性：
1.  **高级初始化（多场景采样策略）**：起点不应固定单一分布，否则模型会过拟合"前 3 步先找地标→第 4 步看到目标"的固定时序。按以下 5 种场景按比例采样，覆盖大尺度导航的全部典型情形：

    | 场景 | 占比 | 起点距离 | 视野状况 | 训练目的 |
    | --- | --- | --- | --- | --- |
    | A. 短距精准对准 | 25% | 距目标 5~15m | 目标已在视野中心附近 | 训练 STOP 决策与精细对准 |
    | B. 中距纯逼近 | 25% | 距目标 30~60m | 目标已在视野内、偏远 | 训练直接抵近能力 |
    | C. 远距探索 | 25% | 距目标 70~120m | 目标视野外、地标可见 | 训练"先飞向地标→再搜索目标"的语义探索 |
    | D. 多地标干扰 | 15% | 距目标 30~80m | 视野内存在 2+ 个同类地标 / 同类目标 | 训练空间词消歧能力 |
    | E. 起点即终点 | 10% | 距目标 ≤1 网格 | 起点直接在目标正上方 | 训练果断 STOP（避免模型不敢停而过冲）|

    **长尾兼容性说明（实施关键）**：本地 DOTA 数据 `small-vehicle` 占 63%，约 60-70% 图像内根本没有 L 类地标，导致场景 C（远距探索）实际可生成的轨迹数远低于 25% 名义比例。批量数据生成时必须：
    - 在 `build_pid_to_path_index` 三件套筛选后，再做一次 **"含 L 类标注"** 的预筛选——仅这类图像可参与场景 C/D 的起点采样；
    - 仅含 T 类的图像只参与场景 A/B/E 的起点采样，且指令构造走 §4.2.2 的 spatial-word 消歧分支；
    - 实际生成完成后**按图像角色分布重算实际占比**，并据此调整下一轮的批量目标量，避免名义比例与实际比例脱节。

    ##### 4.3.1 场景比例动态兜底机制（缺一不可）

    上述"长尾兼容性说明"只是提了方向，但**没有给具体的再平衡算法**。批量生成时必须按以下三步落地，否则实际场景分布会与目标偏差超过 30%（C/D 严重短缺、A/B 严重过剩）。

    **Step 1：扫描全集，计算 L 类图占比上限**
    ```python
    l_class_pids = [pid for pid in train_pids if has_landmark_class(pid)]
    t_only_pids  = [pid for pid in train_pids if pid not in l_class_pids]
    max_cd_ratio = len(l_class_pids) / len(train_pids)   # 例如 0.33
    ```
    实测预估：本地 419 张图（小车主导），`max_cd_ratio` 约在 **0.30~0.40** 之间，远低于名义 `C+D = 0.40`。

    **Step 2：缺额按"B 优先"原则补偿**

    场景 C/D 缺额必须匀给**场景 B（中距纯逼近）**而不是 A 或 E，因为 B 不依赖 L 类地标，但仍能训练"远视野下的语义对准"，与 C 的训练目标最接近（只是起点更近）。
    ```python
    nominal = {'A': 0.25, 'B': 0.25, 'C': 0.25, 'D': 0.15, 'E': 0.10}
    cd_target = nominal['C'] + nominal['D']           # 0.40
    if cd_target > max_cd_ratio:
        shrink = max_cd_ratio / cd_target              # 例如 0.33 / 0.40 = 0.825
        final = {
            'A': nominal['A'],
            'B': nominal['B'] + (cd_target - max_cd_ratio),  # 吸收缺额
            'C': nominal['C'] * shrink,
            'D': nominal['D'] * shrink,
            'E': nominal['E'],
        }
    else:
        final = nominal
    ```

    **Step 3：按 final 比例分别从对应图像池采样**

    - 场景 A / B / E：可从**全部** train_pids 采样（无 L 类要求）
    - 场景 C / D：**只能从 l_class_pids 采样**
    - **采样比例落地**：用 `random.choices(scenarios, weights=[final['A'],...,final['E']])` 决定每条轨迹的场景，再按场景决定图像池——切勿先采样图像再决定场景（会让场景 A/B/E 在 l_class_pids 上过度采样、t_only_pids 闲置）。

    **Step 4：生成完成后输出实际场景占比统计表**
    ```text
    场景实际占比统计:
      A: 24.8% (目标 25.0%)
      B: 32.1% (目标 32.5%)   ← 已吸收 C/D 缺额
      C: 20.5% (目标 20.6%)
      D: 12.4% (目标 12.4%)
      E: 10.2% (目标 10.0%)
    ```
    若偏差 > 3% 需调整下一轮的采样权重。**避免出现"声称按 25/25/25/15/10 采样、实际 38/38/12/6/6"的失实情况**。

    **额外注意**：高度档位 × 场景 = 5×3 = 15 个子分布，每个子分布单独采样。建议先按高度采样再按场景采样（因为高度档位采样比例固定 50/30/20，无视图像类型），最后落到具体图像池。

    每条轨迹随机采样起点方位角 $\phi \in [0°, 360°)$ 与起点 Yaw $\theta_0 \in [0°, 360°)$，避免模型只学到"目标永远在画面上方"。
2.  **贪心寻路循环 (每一步 $t$)**：
    *   **判定阶段（用 local 坐标，与裁剪逻辑一致）**：先用 `world_to_local` 把目标向量转换到当前朝向下的局部系 $(local_x, local_y)$。若 $\max(|local_x|, |local_y|) > \text{fov\_m}/2$ 则目标不在裁剪视野内（注意旋转后视野是正方形，**不能用径向距离 $\sqrt{x^2+y^2}$ 判断**：径向距离落在 $[fov/2,\, fov\cdot\sqrt{2}/2]$ 区间的目标，若位于方框对角方向仍在视野内、若位于方框侧边方向已经出框——径向判定会同时产生"假阴性"与"假阳性"两类错误，污染 EXPLORE / APPROACH 阶段的 CoT 标签）。视野外则当前步骤的临时目标设为**地标**；视野内则设为**最终目标**。
    *   计算当前临时目标在 $(X_t, Y_t)$ 坐标系下的绝对向量 $\vec{v}$。
    *   **Action 计算（理想值）**：将 $\vec{v}$ 转换到当前偏航角 $\theta_t$ 下的相对网格坐标 $(\Delta x, \Delta y)$。若超出了最大步长（如 $\pm 5$），则等比例缩放截断。这就是将要作为大模型 `Action` 标签的值。
    *   **(0,0) 边界处理**：若 round 后 $(\Delta x, \Delta y) = (0, 0)$ 但距离最终目标仍 $> \epsilon$（即"应该对准但不前进"），**整条轨迹该步丢弃**而非强行改成 (0,1)。强行 (0,1) 会让训练数据出现"明明对准了却被迫前进 1 格"的伪标签，污染 STOP 决策学习。该步丢弃只是损失少量样本，不影响总体规模。
    *   **无人机运动学更新（含噪声注入，必须与 evaluate.py 完全一致）**：
        *   **Turn（旋转）— 基于模型 action 累加相对偏航增量，禁止直接对准 active_target 的全局 yaw**：先由截断后的局部网格量 $(\Delta x, \Delta y)$ 推出相对偏航 $\Delta\theta_{local} = \arctan2(\Delta x, \Delta y)$（"右、前"网格量，从 +Y 前轴顺时针测到向量），再累加到当前 yaw 并注入高斯噪声：$\theta_{t+1} = (\theta_t + \Delta\theta_{local} + \mathcal{N}(0, \sigma_\theta)) \bmod 360°$（例如 $\sigma_\theta = 5°$）。
            * **为什么不能用 $\theta_{target}$ 对准（早期方案的坑）**：（1）当 $(\Delta x, \Delta y)$ 被 $\pm 5$ 截断时，"实际飞行方向"会与"模型 action 描述方向"不一致，标签噪声放大；（2）每步强制对准 target 会让训练标签高度偏向 $(0, +N)$，评估时模型只会沿当前朝向直前飞、无法转弯。此修正与 evaluate.py 中 `update_physics_and_draw` 的 `atan2(grid_x, grid_y)` 累加完全一致，保证训练-评估同分布。
        *   **Move（移动）— 沿新 yaw 走，注入距离/横向漂移噪声**：移动距离 $d = \sqrt{\Delta x^2 + \Delta y^2} \cdot \text{grid\_size} + \mathcal{N}(0, \sigma_d)$。在 OpenCV 图像坐标系下（Y 轴向下，正北 = $-Y$ 方向），$\theta_{t+1}$ 对应的位移为 $X_{t+1} = X_t + d \cdot \sin(\theta_{t+1}) + \text{noise}_x$，$Y_{t+1} = Y_t - d \cdot \cos(\theta_{t+1}) + \text{noise}_y$（与代码中 `yaw_step()` 完全一致）。
    *   **图像裁剪（无黑边方案，时序关键）**：图像裁剪发生在每个 step 的**起始**——用 $(X_t, Y_t, \theta_t)$（本 step 起始的位置与朝向，即上一步 Turn-then-Move 物理更新后的状态）切下大图，按 $\theta_t$ 旋转，中心裁剪出目标窗口作为本 step 的模型输入。**严禁先 Turn → Move 再切图**：本 step 的 Turn → Move 产出的 $(X_{t+1}, Y_{t+1}, \theta_{t+1})$ 是**下一 step 的输入状态**，不参与当前 step 的图像裁剪。Step 内执行顺序为：**切图 $(X_t, Y_t, \theta_t)$ → 模型给 action → Turn → Move → 写入 trajectory 数据点**（与 `evaluate.py` 的 `crop_uav_view` → `get_model_action` → `update_physics_and_draw` 完全一致，保证训练-评估同分布；早期"按 $\theta_{t+1}$ 旋转"的表述已修订）。
    *   **终止条件判断**：若**最终目标**距离 $\leq \max(5\text{m},\, 0.5 \times \text{grid\_size})$（与 §1.3 / §3.2 完全一致的统一阈值），记录最终 `Action: STOP` 或 `Action: (0, 0)`（按 80% : 20% 混合采样）并结束。

#### 4.4 脚本逻辑：分阶段规则化思维链 (CoT) 生成
大创项目不需要复杂的语言模型来反向生成 CoT 数据，直接使用简单的**“if-else 规则模版”**即可生成非常标准且符合“探索-抵近”逻辑的思维链。
1.  **阶段 A（探索期，目标不可见）**：当目标距离超出视野范围，但地标可见时。
    *   *CoT 模板*：`Thought: 观察当前图像，目标【{目标类}】尚未出现在视野中，但我看到了地标【{地标类}】在【{地标方位}】。我需要先向地标移动以寻找目标。`
2.  **阶段 B（逼近期，目标可见）**：当目标进入视野范围内时。
    *   *CoT 模板*：`Thought: 观察当前图像，目标【{目标类}】已经出现在视野的【{目标方位}】。目前【{距离描述}】，所以我需要向【{目标方位}】移动。`

3.  **距离描述规则**：
    *   若 $\sqrt{\Delta x^2 + \Delta y^2} > 5$ 网格：赋予“距离较远，尚未到达”。
    *   若 $\leq 5$ 且 $> 1$ 网格：赋予“距离适中，正在接近”。
    *   若 $\leq 1$ 网格：赋予“即将到达目标正上方”。

### 5. 训练策略与完整流程设计
本方案采用**单阶段全量 SFT**：所有字段（2 张图 + 历史动作 + 累计位移 + CoT + Action）一次性训练。早期方案的两阶段 curriculum（Stage 1 mask 字段 + Stage 2 放开）已废弃，原因见 §2.1。

#### 5.1 数据准备阶段
*   **本地数据目录结构（实际形态，脚本必须感知）**：DOTA v2.0 数据在本地按 part 子目录分散存放，且 labelTxt / meta / images 三者命名一一对应但分布于不同根目录：
    ```text
    data/
    ├── train/
    │   ├── images/
    │   │   ├── part4/P{id}.png        (144 张)
    │   │   ├── part5/P{id}.png        (210 张)
    │   │   └── part6/P{id}.png        (65 张)
    │   ├── labelTxt-v2.0/
    │   │   ├── DOTA-v2.0_train/P{id}.txt       (1830 个, 含坐标 + class + difficult)
    │   │   └── DOTA-v2.0_train_hbb/P{id}.txt   (水平框版本, 本项目可忽略)
    │   └── meta/P{id}.txt              (1830 个, 含 imagesource / gsd)
    └── val/
        ├── images/part2/P{id}.png      (135 张)
        ├── labelTxt-v2.0/DOTA-v2.0_val/P{id}.txt  (593 个)
        └── meta/P{id}.txt              (593 个)
    ```
    **关键事实**：图像数 (419/135) 远少于标注数 (1830/593)——这是 DOTA v2.0 发布时 part1-3 单独打包的历史遗留。脚本必须先用 `build_pid_to_path_index` 三方 inner-join (image ∩ label ∩ meta)，仅在齐全的 P-id 上跑后续流程；切勿假设 "labelTxt 全集 = 可用数据全集"。
*   **目标**：使用 Python 脚本生成**约 3,000 ~ 6,000 条轨迹**（对应约 30,000 ~ 60,000 个问答对）。
    *   **规模修正说明（重要事实校正）**：早期目标 5k-10k 轨迹的前提是 DOTA train 有 ~1830 张可用图像。实际本地数据：train 仅 **419 张图**（part4/5/6 合计）和 1830 个 labelTxt（77% 标注无对应图像），val 仅 135 张图 / 593 标注。按 419 张图均摊每张 8-15 条轨迹，叠加 90:10 划分后训练只剩 ~377 张图，无法支撑 5k+ 规模而不严重重复。**实施前必须先 inner-join "有图 ∩ 有标注 ∩ 有 meta" 三件套，再据此推算可达规模**。
*   **输出物**：两个独立 JSONL 文件——`train.jsonl`（参与训练）和 `eval.jsonl`（仅用于验证 loss 与闭环 SR），每条数据都是最完整的最终形态（2 张图 + 历史动作 + 累计位移 + CoT + 最终 Action）。
*   **train/val 划分（防泄露，关键）**：**必须按 DOTA 图像 ID 划分**，不能用 LLaMA-Factory 的 `val_size` 随机切。原因：同一张 DOTA 大图会衍生出多条空间相邻的轨迹（不同起点 / 不同目标），随机切会让验证集和训练集来自同一张大图——模型在训练时已经"见过"该图的视觉特征，验证 loss 会严重偏低，闭环 SR 评估也失真。
    *   正确做法：先在 **DOTA Train 文件夹（即 `data/train/`）内部**按 90% : 10% 划分图像列表 `image_ids_train.txt` / `image_ids_val.txt`，**整张图作为最小划分单位**。
    *   数据生成阶段：分别用两个图 ID 列表生成 `train.jsonl` 与 `eval.jsonl`，两者**绝不共享任何图像**。
    *   YAML 中**不要写 `val_size`**，改用 `eval_dataset: uav_eval` 显式指定，并在 `dataset_info.json` 中同步注册 `uav_eval`。
    *   **评估集双轨定位（重要修订）**：
        - **训练 loss 监控集** = `image_ids_val.txt`（DOTA train 90:10 留出，约 42 张图）。仅用于 `eval_strategy: steps` 的训练期 loss 曲线观察，监控过拟合。**不作为最终 SR 指标依据**——42 张图采样多样性不足，单图遮挡 / 误标会显著扭曲数字。
        - **闭环 SR 主指标集** = DOTA 官方 `data/val/`（135 张图全部三件套齐全，实测）。这是真正独立的留出集（与 DOTA train 完全无图像重叠），**作为最终 SR/SPL 报告的主战场**。早期文档把它定位为"额外 OOD 报告"已修订为"主指标"，更符合 VLN 评估惯例。
        - **报告时仍可双栏对比**：`image_ids_val` (n=42) 与 DOTA val (n=135)，两者差距越小说明泛化越好；但单一主指标采用后者。

#### 5.2 单阶段训练 (Unified SFT)
*   **数据形态**：完整的"2 张图（T-1 + T）+ 5 步历史动作 + 累计位移 → `Thought: xxx \n Action: (x,y)`"。所有字段均为真实值，无 mask、无占位填充。
*   **Vision Encoder 策略**：**ViT 完全冻结，multi-modal projector 全参解冻**。
    *   `freeze_vision_tower: true` —— ViT (~600M 参数) 不动，避免单卡 4090 OOM 且少量数据下劣化视觉特征。
    *   `freeze_multi_modal_projector: false` —— **projector 解冻参与训练（关键修订）**。理由：
        1. projector 是 ViT → LLM 的语义对齐瓶颈层，参数量仅 ~30M（相对 ViT 600M 与 LLM 7B 几乎可忽略），解冻它对显存影响 <5%；
        2. DOTA 卫星图 → 无人机航拍图存在显著 domain gap，projector 是**调整跨模态映射最便宜的手段**，远比解冻 ViT 后段安全；
        3. ViT 视觉特征本身是预训练通用特征（高度泛化），但 projector 把视觉特征映射到 LLM 语义空间，这一步对"DOTA 物体类别 vs 中文短语"的对齐有直接帮助。
    *   背景：LLaMA-Factory 的 YAML 仅支持三个整体冻结开关（`freeze_vision_tower` / `freeze_multi_modal_projector` / `freeze_language_model`），**没有原生的"按层冻结 ViT"参数**（早期方案中的 `vision_tower_finetune_layers` 在 LLaMA-Factory 中不存在，YAML 中写它会被静默忽略）。
    *   仍要进一步缓解 DOTA 卫星图 → 无人机航拍图 domain gap 的扩展路径（**按时间余量自选**）：
        1. **优先选项**：走 §7 的真实数据轻量微调（200-500 条真机数据 + 1:4 仿真混合），让真实数据天然缩小 domain gap。
        2. **进阶选项**（需写额外代码）：自写训练 callback，在 `Trainer` 初始化后手动把 ViT 前 N 层的 `requires_grad=False`、仅留最后 4-8 层可训。学习率必须降到 2e-5 + 加大 warmup。**这不是 YAML 配置项**，需要 monkey-patch 或修改 LLaMA-Factory 源码。
*   **训练目标**：用 LoRA 训练约 **2-3 个 Epoch**。模型需要同时学会：
    1. 看图输出局部网格坐标（跨模态空间映射）
    2. 利用历史动作序列识别"已走了多远 / 是否绕路"（时序推理）
    3. 利用累计位移建立全局位置感（避免无限绕路）
    4. 用 CoT 自然语言解释决策过程
*   **可选 Curriculum（数据采样顺序，不改字段）**：若担心远距探索样本收敛慢，可在单阶段内按 epoch 调整起点场景采样比例：
    *   Epoch 1：A 短距 + E 起点即终点 占 60%（先学会 STOP 与精细对准）
    *   Epoch 2-3：恢复 §4.3 的标准比例（A 25% / B 25% / C 25% / D 15% / E 10%）
    *   这是**数据顺序课程**——prompt 骨架始终不变，模型不会学到"忽略字段"的错误捷径，与早期废弃的字段 mask 课程本质不同。

#### 5.3 自动化评估 (Evaluation)
*   编写一个**推理引擎脚本 (Inference Engine)**，加载微调后的模型。
*   在 `image_ids_val.txt` 对应的 DOTA 图像上，随机初始化起点，让模型连续自回归预测 Action。
*   记录成功率 (Success Rate)、平均步数和 SPL 指标。如果成功率达到 60% 以上（在长距离下），这已经是非常优秀的成果。

#### 5.4 完整训练超参建议

##### 5.4.1 GPU 配置与图像尺寸选择（关键）

Qwen2.5-VL ViT 使用 `patch_size=14`，每张图占用的 visual tokens ≈ `(IMG_SIZE / 14)²`。多模态训练中**图像 token 是显存大头**（远超文本 token），必须根据硬件预算选择 IMG_SIZE。两张图输入下：

| IMG_SIZE | 单图 visual tokens | 两图 visual tokens | 配 cutoff_len 建议 | 适合 GPU |
| --- | --- | --- | --- | --- |
| 224 | 256 | 512 | 2048 | 任意 |
| **336** | **576** | **1152** | **3072** | **单卡 4090 (24GB) ✓** |
| **448** | **1024** | **2048** | **4096** | **双卡 4090 (48GB) ✓** |
| 512 | ~1369 | ~2738 | 4096+ | 不推荐（OOM 风险）|

**推荐配置（与数据生成脚本的 `IMG_SIZE` 必须严格一致，避免重采样浪费）**：

```yaml
# === 单卡 4090 (24GB) - 推荐折中配置 ===
image_max_pixels: 112896           # 336*336 ≈ 576 visual tokens/张
# 数据生成时 IMG_SIZE = 336
per_device_train_batch_size: 1
gradient_accumulation_steps: 8     # 等效 batch_size = 8

# === 双卡 4090 (48GB) - 推荐高质量配置 ===
image_max_pixels: 200704           # 448*448 ≈ 1024 visual tokens/张
# 数据生成时 IMG_SIZE = 448
per_device_train_batch_size: 1
gradient_accumulation_steps: 4     # 等效 batch_size = 8
```

**说明**：
- 单卡 4090 强行用 448 会触发 OOM（即使 grad_ckpt + bf16），实测 336 是 24GB 显存下稳定且视野清晰度足够的最优解。
- 若数据生成 `IMG_SIZE=336` 但训练 `image_max_pixels=200704`（448²），LLaMA-Factory 会**把 336 插值放大到 448**——纯属浪费，必须保持两者数值开方一致。
- 单阶段 SFT 全程保持 `freeze_vision_tower: true` 但 **`freeze_multi_modal_projector: false`**（见 §5.2）。projector 解冻显存增量 <5%，对 DOTA→航拍 domain gap 有显著帮助。若有双卡 48GB 显存余量并尝试解冻 ViT，需注意：显存峰值会再涨约 20%；学习率必须降到 2e-5 并加大 warmup；少量数据下大概率劣化视觉特征，不建议作为默认路径。

##### 5.4.2 完整 YAML 模板（单阶段，以双卡 4090 / IMG_SIZE=448 为例）

**关键点说明**：
*   `bf16` 优于 `fp16`：在 4090/A100 上 bf16 数值范围更大，VLM 多模态训练几乎不溢出
*   `gradient_checkpointing` 必开：7B + 多图训练显存压力大
*   `warmup_ratio: 0.05`：单阶段 SFT 起点是预训练权重，加大 warmup 让 LoRA 平滑进入
*   `num_train_epochs: 3.0`：因为不再有 Stage 1 预热，单阶段需要稍多 epoch 才能收敛
*   `save_total_limit: 3` 避免磁盘塞满
*   `eval_dataset` + 按 DOTA 图划分：**核心防泄露设计**，详见 §5.1
*   监控指标：除 loss 外，建议自定义 metric 监控 **Action token accuracy** 和 **STOP token F1**，这两个直接反映导航能力

### 6. 部署方案：远端推理 + 机载飞控

考虑到 Qwen2.5-VL-7B FP16 显存约 14GB，单步推理 1-3s，绝大部分消费级 / 工业级无人机机载计算单元（Jetson Orin Nano 8GB / Pixhawk）无法本地推理，本项目采用 **"远端推理 + 5G/WiFi 回传，机载只做飞控"** 的方案：

#### 6.1 系统拓扑
```
[无人机机载]                          [地面 GPU 工作站]
摄像头 → 图像预处理(裁剪/Resize)
        ↓ (WiFi 5GHz / 5G / 4G)
        ─────────────────→  Qwen2.5-VL-7B (vLLM / LLaMA-Factory API)
        ←─────────────────  Action (x,y) / STOP
脚本解析 → 旋转 θ 度 + 前进 d 米
        ↓
PX4 / ArduPilot / DJI SDK (MAVLink)
飞控执行 → 悬停拍摄下一帧
```

#### 6.2 关键链路设计
*   **通信**：优先 5GHz WiFi（视距内 200-500m，延迟 ~10ms），超视距用 5G CPE 中继
*   **图像传输**：JPEG 压缩到 80% 质量，单张 ~50KB，2.4Mbps 带宽足够
*   **推理服务**：vLLM 部署 + INT8 量化（精度损失 <2%，吞吐提升 2-3 倍），单 4090 可达 5-10 req/s
*   **超时与心跳**：单次决策超时 ≥5s 触发悬停 + 重试；连续 3 次超时触发返航
*   **决策频率**：受悬停-拍摄-推理-移动的循环节拍限制，约 0.2-0.5 Hz（即 2-5 秒一步），与"悬停拍摄"模式天然契合

#### 6.3 安全护栏（必须实装）
*   **Geofence（电子围栏）**：动作合法性校验，超出预设飞行区域强制 STOP
*   **低电量返航**：电量 <30% 自动切换返航模式
*   **通信丢失保护**：失联 >10s 触发悬停，>30s 触发降落
*   **动作合法性**：飞控侧对模型输出的 (dx, dy) 二次校验（防御 VLM 异常输出如 (99, -99)）

#### 6.4 部署降本备选方案（资源紧张时）
| 方案 | 模型 | 显存 | 单步延迟 | 适用场景 |
| --- | --- | --- | --- | --- |
| A. 远端 FP16 | Qwen2.5-VL-7B | 14GB | 1-3s | 推荐方案（实验室 / 大创演示）|
| B. 远端 INT8 | Qwen2.5-VL-7B (AWQ) | 8GB | 0.6-1.5s | 高吞吐 / 多机协同 |
| C. 蒸馏小模型 | Qwen2-VL-2B (蒸馏+QLoRA) | 4GB | 0.3-0.8s | 远端机器较弱时 |
| D. 机载部署 | Qwen2-VL-2B + INT4 | 2GB | 0.5-1.5s | Jetson Orin NX 16GB 可跑 |

### 7. 真实数据微调（Sim-to-Real 最后一公里）— **Stretch Goal / 视项目时间余量决定**

> **定位说明**：本节为**可选 Stretch Goal**，不属于大创核心交付。是否实施取决于项目时间余量、实机硬件可用性和指导老师的优先级判断。**核心交付（DOTA 仿真数据 + 单阶段 SFT 微调 + 闭环 SR/SPL 评估）已经足以构成完整的大创成果**——本节是"锦上添花"的真机演示部分。
>
> **风险提示（决定是否上马时务必权衡）**：
> - **场景维度不匹配**：DOTA 训练覆盖 500-1000m 大尺度，但学校操场/停车场仅 ~100m，仿真训练的"远距探索"能力可能完全用不上。
> - **采集成本高**：手动遥控记录 (dx, dy, yaw)，单条轨迹 5-10 分钟悬停拍照，500 条 ≈ 50 小时纯采集时间。
> - **标定依赖**：相机内参标定、磁罗盘标定、GPS 标定流程一项都不能少，否则 action 标签噪声极大。
> - **天气与法规**：户外飞行受天气、空管和校园飞行许可限制。
>
> **如果跳过本节的兜底方案**：用 DOTA 仿真飞行 GIF 作为最终演示素材（在闭环 evaluate.py 中可视化），效果同样直观，已经能完整支撑大创答辩。

**如果决定实施**：仅用 DOTA 仿真数据训练出的模型，实机直接零样本部署 SR 通常在 5%-20% 之间（域差距严重）。补充少量真实数据微调可将实机 SR 提升到 50%-70%。

#### 7.1 数据采集（轻量）
*   **采集设备**：你的实机无人机（与最终部署同一台）
*   **场景数量**：3-5 个不同场景（如学校操场、停车场、小区花园），每场景 50-100 条轨迹
*   **总规模**：200-500 条轨迹即可，每条 5-10 步，总样本量约 2000-5000
*   **标注简化**：
    *   不需要精细 CoT，只需 `(图, 任务文本, 正确 action)` 三元组
    *   action 标签的获取：手动遥控飞到正确位置，记录无人机自身位移 (dx, dy)，按 grid_size 离散化即可
    *   可选：用单阶段 SFT 模型先自动标注，人工只校验和修正错误样本（active learning）

#### 7.2 微调策略
*   **加载基座**：§5 单阶段 SFT 训练出的 LoRA 完整 checkpoint（ViT 仍冻结，理由见 §5.2）
*   **学习率**：1e-5（远低于单阶段 SFT 的 2e-5，避免破坏已学好的能力）
*   **Epoch**：1-3 个，过多会过拟合校园场景丢失泛化
*   **数据混合**：建议 **真实数据 : 仿真数据 = 1 : 4** 混合训练，既学新域又防遗忘
*   **冻结策略**：仅训 LLM 端 LoRA（`freeze_vision_tower: true`），避免少量数据破坏视觉特征

#### 7.3 验证流程
1.  在真实场景的留出集（held-out test set，至少 30 条轨迹）评估 SR / SPL
2.  对比"仅仿真训练"vs"仿真+真实微调"的 SR 提升幅度，作为大创成果的关键证据
3.  生成实机飞行的真实视频和轨迹叠加图，作为最终展示素材
